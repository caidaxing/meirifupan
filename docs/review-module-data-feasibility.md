# 复盘模块数据可行性与缺口评估

本文在《复盘模块实施方案》的基础上，评估每个子模块需要的数据当前是否具备、缺什么、怎么补。本文不改变产品目标，只用于安排开发顺序。

## 1. 总体结论

复盘模块可以开始开发，但要分阶段。

当前适合先做：

| 模块 | 结论 |
| --- | --- |
| 涨停原因 | 可以做，已有核心数据 |
| 涨停梯队 | 可以做，已有核心数据 |
| 晋级 | 可以做基础版，细节反馈需要增强 |
| 题材轮动 | 可以做，已有独立数据 |
| 龙虎榜 | 可以做基础版，席位明细需要增强 |
| 异动 | 可以做普通异动基础版，交易所异动提醒需要补 |
| 涨幅梯队 | 需要补数据后做完整版 |

开发建议：

```text
先做页面框架和能用已有数据支撑的 5 个模块。
涨幅梯队和交易所异动作为第二步补数据。
```

## 6. 涨幅梯队

### 6.1 需要的数据

| 字段 | 当前状态 | 说明 |
| --- | --- | --- |
| 全市场股票列表 | 部分缺 | 当前不保证完整 |
| 10/20/30 日 K 线 | 缺口较大 | `stock_kline_daily` 只覆盖核心股 |
| 区间起始价 | 缺口较大 | 需要全市场或候选池 K 线 |
| 区间结束价 | 缺口较大 | 需要收盘价或快照 |
| 区间涨幅 | 可计算 | 依赖 K 线 |
| 题材标签 | 部分已有 | 涨停股题材有，非涨停强势股缺 |
| 市场标识 | 部分缺 | 可由代码推断，最好标准化 |

### 6.2 可先做版本

可以先做占位和接口结构，但完整页面需要补数据。

第一版可选两条路：

| 方案 | 说明 |
| --- | --- |
| 接 Quantzz `/v3/market/pct-tier` | 快速得到涨幅梯队 |
| 用扶摇 K 线自算 | 更可控，但需要全市场 K 线或候选池 |

### 6.3 推荐

第一步先接 Quantzz 涨幅梯队接口，快速把页面跑起来。第二步再用扶摇 K 线沉淀本地计算能力。

## 10. 异动

### 10.1 两类异动要区分

| 类型 | 含义 | 当前状态 |
| --- | --- | --- |
| 普通盘中异动 | 拉升、跳水、大单、封板等 | 已有 `movement_alerts` |
| 交易所异动提醒 | 10/30 日涨幅偏离触发风险 | 缺失 |

Quantzz 的复盘异动页更接近“交易所异动提醒”。当前项目已有的是普通盘中异动。

### 10.2 普通异动需要的数据

### 10.3 交易所异动需要的数据

## 11. 数据补齐优先级

### P0：不补也能先开工

| 数据 | 支撑模块 |
| --- | --- |
| 涨停事件 | 涨停原因、涨停梯队、晋级 |
| 涨停题材映射 | 涨停原因、涨停梯队、题材轮动 |
| 题材轮动表 | 题材轮动 |
| 龙虎榜股票级表 | 龙虎榜基础版 |
| 普通异动表 | 异动基础版 |

### P1：建议马上补

| 数据 | 原因 |
| --- | --- |
| 标准 `thscode` | 后续接扶摇和统一行情必须要 |
| 实时/收盘行情快照 | 晋级反馈、涨幅梯队、龙虎榜增强都需要 |
| 更完整个股 K 线 | 涨幅梯队和交易所异动需要 |

### P2：后续补

| 数据 | 原因 |
| --- | --- |
| 席位明细和席位标签 | 龙虎榜增强 |
| 交易所异动规则 | 异动增强 |
| 板块指数 K 线 | 题材轮动增强 |
| 分时数据 | 分时小图和盘中观察 |

## 12. 第一阶段开发范围

第一阶段范围定为：

```text
复盘模块 7 子 Tab
涨停原因基础版
涨停梯队基础版
晋级基础版
题材轮动基础版
龙虎榜基础版
普通异动基础版
涨幅梯队占位 + 数据缺口提示
```

第一阶段不做完整涨幅梯队和交易所异动精确计算，因为这两个需要补全市场 K 线和规则。

## 13. 开发前剩余缺口

文档层面已补齐后，可以开始第一阶段开发。

仍需在开发中处理的技术点：

1. 定义新的复盘聚合接口。
2. 增加前端 7 子 Tab。
3. 统一 TypeScript 类型。
4. 为后端聚合函数补测试。
5. 每个页面缺字段时显示 `warnings`，不能假填。
