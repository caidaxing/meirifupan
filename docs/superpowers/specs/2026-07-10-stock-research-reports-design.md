# 个股研报采集与展示设计

## 1. 目标

在现有 A 股复盘项目中新增个股研报数据能力，首版完成以下闭环：

- 从东方财富个股研报公开接口采集最近 30 个自然日的数据。
- 保存研报基础信息、分析师、盈利预测、详情摘要和接口原始数据。
- 下载并长期保留每篇研报的 PDF 原文，不只保存外部链接。
- 提供查询和本地 PDF 访问接口。
- 将“新闻资讯”中的“研报线索”占位升级为可用的“个股研报”页面。
- 支持后续每日增量更新，并能对失败项目重复补抓。

首版不包含行业研报、策略报告、宏观研究、券商晨会、研报智能解读和研报对股价影响回测。

## 2. 数据来源

### 2.1 列表接口

- 请求地址：`https://reportapi.eastmoney.com/report/list2`
- 请求方式：`POST application/json`
- 每页数量：50 条。
- 按开始日期和结束日期分页采集，直到达到接口返回的总页数。
- 研报唯一标识使用 `infoCode`，例如 `AP202607101826859348`。

列表可取得股票、机构、分析师、评级、评级变化、行业、目标价、附件页数、附件大小和未来三年 EPS/PE 等字段。

### 2.2 详情页

- 页面地址：`https://data.eastmoney.com/report/info/{info_code}.html`
- 页面脚本中的 `zwinfo` 保存研报详情摘要和 PDF 地址。
- 使用 HTML 脚本节点解析加 `json.JSONDecoder` 读取对象，避免依赖整页字符串替换。

详情数据包含摘要正文、发布时间、研报标题、来源机构、分析师、股票关系、PDF 地址和附件信息。

### 2.3 PDF

- PDF 必须下载到本地，不以外部链接作为完成状态。
- 文件位置：`data/research_reports/YYYY/MM/DD/{info_code}.pdf`。
- 数据库只保存相对路径，保证本地和服务器目录可迁移。

## 3. 数据结构

首版使用四张表，避免把重复列表和多值字段塞进一张大表。

### 3.1 `stock_research_reports`

保存一篇研报一条核心索引。

| 字段 | 说明 |
| --- | --- |
| `info_code` | 主键，东方财富研报编号 |
| `publish_date` | 发布日期时间 |
| `stock_code`、`stock_name` | 关联股票 |
| `market` | 上市市场 |
| `title` | 研报标题 |
| `org_code`、`org_name`、`org_short_name` | 发布机构 |
| `industry_code`、`industry_name` | 个股所属行业 |
| `rating_name`、`previous_rating_name` | 当前和上次评级 |
| `rating_change_code`、`rating_change_name` | 来源代码及归一化后的首次、维持、调高、调低或无 |
| `target_price_low`、`target_price_high` | 目标价区间 |
| `source_url` | 东方财富详情页 |
| `detail_status` | `pending`、`fetched` 或 `failed` |
| `raw_payload` | 列表接口原始 JSON |
| `created_at`、`updated_at` | 本地维护时间 |

`stock_code` 同时写入现有 `stocks` 表，研报通过股票代码与行情、公告和复盘数据关联。

### 3.2 `stock_research_report_authors`

保存一篇研报对应的多个分析师。

主键为 `(info_code, author_id)`，字段包括 `author_name` 和 `sort_order`。

### 3.3 `stock_research_report_forecasts`

按预测年度保存 EPS 和 PE，不使用“今年、明年、后年”固定列。

主键为 `(info_code, forecast_year)`，字段包括 `eps` 和 `pe`。接口返回的 `currentYear` 用于把相对年度转换为明确年份。

### 3.4 `stock_research_report_contents`

保存详情摘要和 PDF 下载状态。

| 字段 | 说明 |
| --- | --- |
| `info_code` | 主键并关联研报索引 |
| `summary_text` | 详情页中的研报摘要正文 |
| `pdf_url` | 来源 PDF 地址 |
| `local_pdf_path` | 本地 PDF 相对路径 |
| `pdf_status` | `pending`、`downloaded` 或 `failed` |
| `attach_pages` | 来源声明的 PDF 页数 |
| `declared_pdf_size_kb` | 来源声明的附件大小，单位 KB |
| `pdf_size` | 本地文件字节数 |
| `pdf_sha256` | 文件 SHA-256 |
| `pdf_error` | 最近一次失败原因 |
| `downloaded_at` | 下载完成时间 |
| `raw_payload` | 详情页 `zwinfo` 原始 JSON |
| `created_at`、`updated_at` | 本地维护时间 |

## 4. 采集流程

### 4.1 首轮回补

1. 以执行日为结束日期，向前回补 30 个自然日。
2. 分页请求列表接口，并按 `info_code` 幂等写入索引、分析师和盈利预测。
3. 对详情未完成的研报抓取详情页并保存 `zwinfo` 原始数据。
4. 将 PDF 下载到同目录下的 `.part` 临时文件。
5. 校验 HTTP 状态、`Content-Type`、`%PDF-` 文件头和非零大小；将来源 `attach_size` 按 KB 换算，允许因取整产生最多 2 KB 差异。
6. 计算 SHA-256 后原子改名为正式 `.pdf` 文件。
7. 更新详情状态和 PDF 状态，记录成功、失败和错误原因。
8. 将本次任务结果写入现有 `data_jobs`，包括列表数、详情成功数、PDF 成功数和失败数。

### 4.2 日常增量

- 每日 `07:30` 和 `19:30` 各执行一次。
- 每次请求最近 2 个自然日，处理当天新研报并补抓前一次失败项。
- 数据以 `info_code` 幂等更新，不因接口暂时返回空列表删除已有记录。
- 同一个已经校验成功的 PDF 不重复下载；缺失、大小不符或状态失败时允许重试。

### 4.3 请求控制

- 列表请求串行分页。
- 详情和 PDF 使用小并发，默认并发数为 3。
- 单次请求超时 20 秒，失败最多重试 3 次并使用递增等待。
- 单篇失败不终止整个批次。

## 5. 后端接口

### 5.1 日期

`GET /api/research-reports/dates`

返回已有研报日期，新日期在前。

### 5.2 列表

`GET /api/research-reports?date=YYYY-MM-DD&q=&rating=&org=&limit=200`

支持按日期、股票代码、股票名称、标题、评级和机构筛选。响应包含总数、返回数、评级分组、机构分组、PDF 成功数和列表项目。

### 5.3 详情

`GET /api/research-reports/{info_code}`

返回核心信息、分析师、年度盈利预测、摘要正文、PDF 状态和本地 PDF 访问地址。

### 5.4 本地 PDF

`GET /api/research-reports/{info_code}/pdf`

只读取数据库中已确认下载成功的本地文件，并使用 `FileResponse` 以内联 PDF 返回。文件不存在、校验未完成或路径越界时返回明确错误，不回退到外部 PDF 地址。

## 6. 前端设计

### 6.1 模块位置

保留现有一级“新闻资讯”模块，将其中“研报线索”Tab 改为“个股研报”。不新增一级侧栏，也不把研报混入公告列表。

### 6.2 页面结构

模块切换仍在最上方，筛选和数量信息位于 Tab 下方：

1. 顶部：公告速递、7x24 快讯、政策日历、海外隔夜、个股研报。
2. 控制区：研报数量、研报日期、关键词、评级、机构。
3. 内容区左侧：研报列表。
4. 内容区右侧：所选研报详情。

列表展示发布日期、股票代码和名称、标题、机构、评级、评级变化、行业及 PDF 状态。

详情展示摘要、分析师、目标价、年度 EPS/PE、附件页数和大小，并提供“查看本地 PDF”按钮。按钮打开同源的本地 PDF 接口，不跳转东方财富 PDF 地址。

### 6.3 响应式行为

- 桌面端使用列表和详情双栏布局。
- 移动端改为单栏，列表在前，选择研报后详情紧随其后。
- 搜索框和选择器不使用固定大高度，避免移动端出现异常空白。
- PDF 在新页面使用浏览器原生查看器打开，避免移动端内嵌 PDF 造成滚动和宽度问题。

## 7. 失败处理与可追溯性

- 列表原始 JSON 和详情原始 JSON 都保留在数据库。
- PDF 失败原因写入详情表，后续任务可以继续补抓。
- `.part` 文件不会被页面接口当作可用 PDF。
- 写数据库和移动正式文件按单篇研报完成，程序中断后可从未完成状态续跑。
- 接口和页面区分“详情未取回”“PDF 下载失败”和“本地文件缺失”。
- 页面不因单篇 PDF 失败而隐藏该研报基础信息。

## 8. 验证

### 8.1 自动测试

- 列表响应字段映射、分页和空页测试。
- `zwinfo` 详情解析测试。
- 数据库建表、幂等写入、分析师和盈利预测关联测试。
- PDF 文件头、大小、SHA-256、临时文件和失败重试测试。
- 列表、详情和本地 PDF API 测试。
- 前端筛选、空状态、错误状态和详情切换测试。

### 8.2 实际数据验证

- 运行最近 30 天回补并记录总研报数、详情成功数、PDF 成功数和失败清单。
- 随机抽查至少 5 篇研报，核对股票、机构、评级、摘要和 PDF 内容。
- 确认数据库重复执行后数量不异常增长。
- 运行完整 Python 测试、前端构建和数据库完整性检查。
- 使用桌面和移动尺寸截图检查筛选区、列表、详情和 PDF 打开流程。

## 9. 完成标准

- 最近 30 天接口可见的个股研报均已写入数据库。
- 每篇研报都有明确的详情状态和 PDF 状态。
- 下载成功的 PDF 均存在于本地、文件头正确并有 SHA-256。
- 页面可以按日期和条件查看研报并打开本地 PDF。
- 每日两次增量任务已接入现有调度器。
- 采集失败可重跑，已有数据不会因单次源站失败被删除。
- 自动测试、前端构建、数据库完整性检查和响应式页面检查通过。
