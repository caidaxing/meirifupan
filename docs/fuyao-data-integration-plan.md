# 扶摇数据接入与落地文档

本文记录扶摇数据、Quantzz、AkShare 在当前项目里的分工，以及后续每一步开发需要做什么、产出什么、留下什么文档。

当前判断：扶摇更适合做稳定的数据底座，Quantzz 继续负责短线复盘里的语义数据，AkShare 作为备用和补充。

## 1. 背景和目标

### 1.1 当前项目目标

项目现在是一个 A 股短线复盘系统，核心是每天收盘后把市场环境、涨停结构、题材强弱、情绪周期、盘前指引整理成一个可看的页面。

当前已经有：

| 能力 | 当前状态 |
| --- | --- |
| 涨停复盘 | 已有涨停原因、涨停梯队、热门板块、高标股、全部涨停明细 |
| 情绪复盘 | 已有情绪趋势、情绪热度、空间板、亏钱效应 |
| 赚钱效应 | 已有封板质量、晋级率、热门股、部分资金反馈 |
| 盘前指引 | 已有隔夜新闻、公告、美股映射、次日策略 |
| 数据总览 | 已有数据任务状态、部分数据覆盖情况 |
| 题材轮动 | 已有后端数据和接口，前端可继续加强 |

### 1.2 为什么要接扶摇

现有数据源能支撑盘后复盘，但在这些地方还不够稳：

| 问题 | 当前情况 | 扶摇可以补什么 |
| --- | --- | --- |
| 实时行情没有统一入口 | 有 AkShare 快照和 Quantzz 实时接口，但没有统一后端 API | 扶摇提供 A 股行情快照 |
| 股票代码不统一 | 项目里同时出现 `600519`、`600519.SH`、股票名 | 扶摇提供标的检索和 `thscode` 标准代码 |
| 历史 K 线依赖分散 | 现在主要靠 AkShare | 扶摇提供标准历史 K 线，支持复权 |
| 财务数据缺失 | 当前项目基本没有财务层 | 扶摇提供利润表、资产负债表、现金流量表 |
| 指数和成分股不足 | 题材、指数、成分关系还不完整 | 扶摇提供同花顺指数列表和成分股 |
| 数据源健康不可见 | 只有任务是否完成，没有源级别状态 | 接入时可以增加数据源日志和健康检查 |

### 1.3 最终目标

第一阶段不是重做系统，而是把底层数据源理顺：

1. 扶摇负责基础行情、代码映射、K 线、交易日历、财务、指数成分。
2. Quantzz 负责涨停原因、题材热度、情绪、人气、短线复盘语义数据。
3. AkShare 保留为备用源，避免某个接口失败时整条链路断掉。
4. 后端统一输出项目自己的 API，前端不需要知道数据来自哪个外部平台。

## 3. 数据源分工

### 3.1 扶摇负责的数据

| 数据 | 接口 | 项目用途 | 优先级 |
| --- | --- | --- | --- |
| 股票搜索和代码映射 | `/api/meta/tickers/search` | 统一 `ticker`、`thscode`、股票名 | 高 |
| 实时行情快照 | `/api/a-share/prices/snapshot` | 盘中首页、涨幅梯队、观察池实时状态 | 高 |
| 历史 K 线 | `/api/a-share/prices/historical` | 个股日 K、核心股复盘、涨幅趋势 | 高 |
| 交易日历 | `/api/a-share/calendar/trading-days` | 调度、日期选择、回补边界 | 中 |
| 财务报表 | `/api/a-share/financials/*` | 后续基本面、盈利质量、风险过滤 | 中 |
| 同花顺指数列表 | `/api/a-share-index/catalog/ths-index-list` | 行业、概念、地域、特色指数目录 | 中 |
| 指数成分股 | `/api/a-share-index/constituents/ths-stock-list` | 题材内股票、板块钻取 | 中 |
| 涨跌停功能数据 | 文档中的同花顺功能数据 | 补充涨停池、连板天梯 | 中 |

### 3.2 Quantzz 继续负责的数据

| 数据 | 保留原因 |
| --- | --- |

| 热门板块 | 已能支撑题材强弱判断 |
| 情绪热度 | 已聚合短线指标，适合复盘 |
| 人气榜 | 用于观察市场关注度 |

### 4.2 半支持

| 模块 | 当前情况 | 需要补什么 |
| --- | --- | --- |
| 实时行情 | 有来源，但没有统一后端 API | 接扶摇快照，统一输出 `/api/market/realtime` |
| 个股分时 | 代码里有方法，但没有稳定进入流水线 | 后续做盘中页面时再接 |
| 个股资料 | 有部分快照，覆盖不完整 | 用扶摇标的检索和列表补齐 |
| 板块原因 | 只覆盖部分重点板块 | 继续用 Quantzz，必要时补缓存 |
| 题材内股票 | 只覆盖热门题材 | 用扶摇指数成分股补全 |
| 人气对比 | 有人气排名，没有多日对比结构 | 增加历史排名表或聚合接口 |
| 涨幅梯队 | 当前不完整 | 用扶摇全市场快照计算 |
| 数据源健康 | 只有任务状态 | 增加每个数据源的成功、失败、耗时、字段缺失记录 |

### 4.3 缺失

| 缺失项 | 建议补法 |
| --- | --- |
| 扶摇 REST API 客户端 | 新增 `src/fuyao_client.py` |
| 统一实时行情接口 | 新增后端 `/api/market/realtime` |
| 全市场实时快照缓存 | 新增 `market_realtime_snapshot` 或轻量缓存表 |
| 标准代码映射 | 扩展 `stocks` 表，加入 `thscode`、`exchange`、`asset_type` |
| 财务数据 | 新增三张财务表，先按核心观察股采集 |
| 指数成分股 | 新增指数目录和成分股表 |
| WebSocket 推送 | 第二阶段再做，第一阶段先轮询 |
| 盘中观察池状态 | 需要实时行情和观察池表配合 |

## 5. 分步落地计划

每一步都需要留下文档。建议所有开发记录都放在 `docs/plans/` 下，每个文件名带日期，方便回看。

### 第 1 步：建立扶摇客户端

目标：先把扶摇 REST 请求封装起来，让后续代码不要到处拼 URL、拼请求头。

开发内容：

1. 新增 `src/fuyao_client.py`。
2. 从环境变量读取 `FUYAO_API_KEY`。
3. 统一处理 `code != 0` 的业务错误。
4. 给每次请求记录 endpoint、耗时、错误信息。
5. 先实现 `search_tickers()` 和 `get_snapshot()` 两个方法。

验收标准：

| 检查项 | 标准 |
| --- | --- |
| 鉴权 | 不在代码中出现真实 API Key |
| 股票搜索 | 能通过“贵州茅台”查到 `600519.SH` |
| 行情快照 | 能查到单只和多只股票 |
| 错误处理 | key 缺失、接口失败时有清楚报错 |

文档产出：

| 文件 | 内容 |
| --- | --- |
| `docs/plans/YYYY-MM-DD-fuyao-client.md` | 记录客户端方法、环境变量、错误处理方式、测试结果 |

### 第 2 步：统一股票代码和名称

目标：解决项目里股票代码格式混乱的问题。

开发内容：

1. 检查 `stocks` 表当前字段。
2. 给股票基础信息补充 `thscode`、`ticker`、`exchange`、`asset_type`、`currency`。
3. 写一个同步脚本，把当前数据库里的股票逐批通过扶摇解析。
4. 对解析失败的股票单独记录，不要静默跳过。

验收标准：

| 检查项 | 标准 |
| --- | --- |
| 代码映射 | 核心股票都有 `thscode` |
| 兼容旧代码 | 旧的 `stock_code` 查询不受影响 |
| 失败记录 | 解析失败列表可查看 |

文档产出：

| 文件 | 内容 |
| --- | --- |
| `docs/plans/YYYY-MM-DD-stock-code-standard.md` | 记录表结构变化、迁移脚本、失败样例、回滚办法 |

### 第 3 步：接入实时行情快照

目标：给盘中页面和涨幅梯队提供统一实时行情。

开发内容：

1. 用扶摇 `/api/a-share/prices/snapshot` 拉取指定股票快照。
2. 支持全市场分页拉取，但第一版先做手动触发或短缓存。
3. 后端新增 `/api/market/realtime`。
4. 支持传 `symbols=600519.SH,000001.SZ`。
5. 返回字段统一成前端好用的格式。

建议返回字段：

| 字段 | 说明 |
| --- | --- |
| `thscode` | 标准代码 |
| `ticker` | 纯股票代码 |
| `name` | 股票名，从本地 `stocks` 表补 |
| `last_price` | 最新价 |
| `change_pct` | 涨跌幅 |
| `volume` | 成交量 |
| `turnover` | 成交额 |
| `updated_at` | 本地更新时间 |

验收标准：

| 检查项 | 标准 |
| --- | --- |
| 单股查询 | 能查一只股票 |
| 多股查询 | 能查观察池股票 |
| 名称补全 | 行情结果能显示中文名 |
| 前端可用 | 前端不需要直接请求扶摇 |

文档产出：

| 文件 | 内容 |
| --- | --- |
| `docs/plans/YYYY-MM-DD-realtime-market-api.md` | 记录接口设计、请求参数、响应字段、缓存策略 |

### 第 4 步：接入历史 K 线

目标：让核心股、观察股、涨停股可以稳定展示日 K。

开发内容：

1. 用扶摇 `/api/a-share/prices/historical` 拉取日 K。
2. 支持 `none`、`forward`、`backward` 三种复权。
3. 先写入现有 `stock_kline_daily`，不够再扩表。
4. 优先采集核心股、涨停股、人气股，不一开始全市场铺开。

验收标准：

| 检查项 | 标准 |
| --- | --- |
| 单股 K 线 | 能拉最近一年日 K |
| 复权 | 默认前复权，参数可切换 |
| 入库 | 能写入 `stock_kline_daily` |
| 回补 | 支持指定股票和日期区间 |

文档产出：

| 文件 | 内容 |
| --- | --- |
| `docs/plans/YYYY-MM-DD-fuyao-kline.md` | 记录 K 线字段映射、复权规则、采集范围、测试股票 |

### 第 5 步：替换或增强交易日历

目标：让调度和回补统一使用可靠交易日历。

开发内容：

1. 用扶摇 `/api/a-share/calendar/trading-days` 同步近一年交易日。
2. 保留 Quantzz 交易日历作为备用。
3. 日常任务先查本地 `trade_calendar`，缺失时再拉外部接口。

验收标准：

| 检查项 | 标准 |
| --- | --- |
| 交易日同步 | 能写入近一年交易日 |
| 日期选择 | 前端日期列表正常 |
| 调度 | 非交易日不误跑核心采集 |

文档产出：

| 文件 | 内容 |
| --- | --- |
| `docs/plans/YYYY-MM-DD-trade-calendar.md` | 记录主备来源、同步周期、非交易日处理 |

### 第 6 步：接入指数目录和成分股

目标：补齐题材、行业、概念指数和成分股关系。

开发内容：

2. 拉取重点指数的成分股。
3. 新增或扩展指数表和成分股关系表。
4. 前端题材钻取时可以展示板块内股票。

验收标准：

| 检查项 | 标准 |
| --- | --- |
| 指数目录 | 能看到概念、行业等分类 |
| 成分股 | 能查询某个指数下有哪些股票 |
| 页面联动 | 题材页面能钻到股票列表 |

文档产出：

## 9. 开发顺序建议

建议按这个顺序做：

1. `FuyaoClient`
2. 股票代码映射
3. 实时行情后端接口
4. 历史 K 线接入
5. 交易日历主备切换
6. 数据源健康检查
7. 指数目录和成分股
8. 财务数据
9. 前端盘中首页和题材钻取

原因很简单：先把“能稳定拿到数据”做出来，再做页面。否则页面先做完，后面数据字段一变，还要来回改。

## 12. 结论

扶摇适合作为当前项目的基础数据主源，先接 REST API，不急着把 MCP 放进服务器业务代码。

落地时优先做：

1. 股票代码标准化。
2. 实时行情统一接口。
3. 历史 K 线。
4. 数据源健康检查。

Quantzz 不替换，继续保留它在短线复盘里的价值。AkShare 也不删，作为备用源和特色数据补充。
