# Quantzz 同类产品复刻方案

本文回答一个问题：如果要把 `quant.zizizaizai.com` 这一类短线量化复盘网站做出来，应该怎么做。

这里说的“复刻”不是照抄页面、代码、品牌或登录后的私有数据，而是复刻它的核心能力：盘中看情绪，盘后做复盘，盘前给指引，把短线交易最关心的数据整理成可读、可查、可追踪的系统。

## 1. 总体判断

当前项目已经有一个不错的基础，不需要从零开始。

已经具备：

| 能力 | 当前情况 |
| --- | --- |
| 后端服务 | 已有 FastAPI |
| 前端页面 | 已有 React/Vite 看板 |
| 数据库 | 已有 SQLite 表结构和多张复盘表 |
| 定时任务 | 已有盘前、盘后任务 |
| 涨停复盘 | 已经接近 Quantzz 的核心模块 |
| 情绪复盘 | 已有趋势和热度基础 |
| 题材轮动 | 已有后端能力，前端可继续加强 |
| 盘前指引 | 已经有新闻、公告、海外映射 |

真正缺的是三件事：

1. **盘中能力不够**：实时行情、盘中情绪、异动时间线还没做成完整页面。
2. **数据源层不统一**：Quantzz、AkShare、扶摇的数据还没有统一封装和主备策略。
3. **产品结构还没完全成型**：已有页面偏“数据展示”，下一步要变成“看盘和复盘工作台”。

## 2. 产品方案

### 2.1 产品定位

产品定位：A 股短线交易者的盘中观察和盘后复盘系统。

一句话：每天把市场情绪、涨停结构、题材主线、人气核心和明日观察方向整理清楚。

目标不是替用户做买卖决策，而是解决三个问题：

| 问题 | 产品回答 |
| --- | --- |
| 今天市场强不强 | 情绪、涨跌家数、涨停跌停、炸板、指数 |
| 主线在哪里 | 涨停原因、题材分组、题材轮动、核心股 |
| 明天看什么 | 盘后总结、盘前指引、观察池、风险点 |

### 2.2 目标用户

| 用户 | 需求 |
| --- | --- |
| 短线交易者 | 快速看清涨停结构、题材强弱、接力情绪 |
| 复盘型投资者 | 每天形成可追踪的市场记录 |
| 半自动化量化用户 | 希望把行情、情绪、题材数据沉淀下来 |
| 自用投研系统用户 | 不想依赖多个网页来回翻 |

### 2.3 产品模块

建议按 8 个主模块组织。

| 模块 | 页面内容 | 解决什么问题 | 第一阶段 |
| --- | --- | --- | --- |
| 首页总览 | 指数、涨跌家数、情绪、涨停跌停、核心题材 | 一眼判断市场状态 | 做 |
| 涨停复盘 | 涨停原因、题材分组、连板梯队、全部涨停股 | 看主线和短线高度 | 做 |
| 情绪周期 | 情绪热度、空间板、亏钱效应、多日趋势 | 判断周期位置 | 做 |
| 赚钱效应 | 封板率、晋级率、炸板、跌停、热门股反馈 | 判断是否适合接力 | 做 |
| 题材轮动 | 多日题材强度、题材趋势、题材内核心股 | 看主线延续或切换 | 做 |
| 盘中观察 | 实时行情、涨幅梯队、异动时间线、人气变化 | 盘中盯盘 | 第二阶段 |
| 盘前指引 | 隔夜新闻、公告、美股映射、今日观察方向 | 开盘前准备 | 做 |
| 数据总览 | 数据任务、数据源健康、缺失字段 | 判断数据是否可靠 | 做 |

### 2.4 首页设计

首页不要做成宣传页，直接进入看盘工作台。

首页区域：

| 区域 | 内容 |
| --- | --- |
| 市场状态 | 上证、深成指、创业板、北证 50 |
| 情绪仪表 | 情绪热度、自然涨停、跌停、炸板、最高板 |
| 赚钱效应 | 上涨家数、下跌家数、平均涨幅、成交额 |
| 今日主线 | 涨停数量最多、强度最高的题材 |
| 核心股票 | 高标股、人气股、题材龙头 |
| 风险提示 | 跌停、炸板、昨日高标反馈 |
| 任务状态 | 今日数据是否已更新 |

## 3. 技术方案

### 3.1 总体架构

建议保持当前技术栈，逐步增强。

```text
外部数据源
  -> 数据采集层
  -> 数据清洗和标准化层
  -> SQLite/PostgreSQL
  -> FastAPI
  -> React 前端
  -> 定时任务和数据源健康检查
```

### 3.2 技术选型

| 层 | 建议 | 原因 |
| --- | --- | --- |
| 前端 | React + Vite | 当前已经在用，开发快 |
| 图表 | ECharts | 适合行情图、热力图、趋势图 |
| 后端 | FastAPI | 当前已经在用，接口清晰 |
| 数据库第一阶段 | SQLite | 当前已有，迁移成本低 |
| 数据库第二阶段 | PostgreSQL | 线上长期运行更稳 |
| 缓存 | Redis 或内存 TTL 缓存 | 盘中实时行情需要短缓存 |
| 调度 | APScheduler 或 cron | 当前已有调度基础 |
| 部署 | Docker Compose | 当前服务器环境已经支持 |
| 日志 | 标准日志 + 数据源运行表 | 方便排查采集问题 |

### 3.3 后端模块划分

建议把后端拆成这些边界：

| 模块 | 职责 |
| --- | --- |
| `clients/` | 外部数据源客户端，封装 Quantzz、Fuyao、AkShare |
| `collectors/` | 采集任务，负责拉取和入库 |
| `normalizers/` | 字段清洗、代码标准化、日期转换 |
| `repositories/` | 数据库读写 |
| `services/` | 业务聚合，比如情绪、题材、复盘 |
| `api/` | FastAPI 路由 |
| `scheduler/` | 定时任务 |

当前项目不一定马上改目录，但新功能按这个边界写，避免继续堆到一个文件里。

| `/api/stocks/{thscode}/kline` | 个股 K 线，待新增 |

需要新增或增强：

| 表 | 用途 |
| --- | --- |

### 3.7 部署方案

当前服务器可以继续用 Docker Compose。

建议服务：

```text
app        FastAPI + 前端静态文件
scheduler 盘前、盘后、盘中采集任务
db         第一阶段 SQLite 文件挂载，第二阶段 PostgreSQL
redis      第二阶段用于实时缓存
nginx      反向代理和 HTTPS
```

第一阶段不急着引入太多服务，先把业务跑通。

| 题材热度 | Quantzz | 本地涨停映射计算 | 题材主线 |
| 情绪热度 | Quantzz | 本地涨停、跌停、炸板计算 | 情绪周期 |
| 人气榜 | Quantzz/同花顺 | 东方财富、AkShare | 市场关注度 |
| 跌停池 | AkShare | 扶摇功能数据，如可用 | 亏钱效应 |
| 炸板池 | AkShare | 本地盘中事件计算 | 封板质量 |
| 龙虎榜 | AkShare | 其他公开源 | 资金行为 |

| 财务数据 | 扶摇 | AkShare | 后续基本面 |
| 新闻公告 | 财联社、东方财富、AkShare | 新浪、央视等 | 盘前指引 |

### 4.2 扶摇数据怎么用

扶摇适合作为基础数据主源。

优先接入：

| 步骤 | 接口 | 目的 |
| --- | --- | --- |
| 1 | `/api/meta/tickers/search` | 统一股票代码 |
| 2 | `/api/a-share/prices/snapshot` | 实时行情 |
| 3 | `/api/a-share/prices/historical` | 历史 K 线 |
| 4 | `/api/a-share/calendar/trading-days` | 交易日历 |
| 5 | `/api/a-share-index/catalog/ths-index-list` | 指数目录 |
| 6 | `/api/a-share-index/constituents/ths-stock-list` | 成分股 |
| 7 | `/api/a-share/financials/*` | 财务数据 |

使用方式：

```http
X-api-key: ${FUYAO_API_KEY}
```

API Key 只放服务器环境变量，不写进代码和文档。

### 4.3 Quantzz 数据怎么用

Quantzz 适合保留短线语义数据。

保留接口：

| 接口 | 用途 |
| --- | --- |
| `/v3/api/review/uplimit/reason` | 涨停原因 |
| `/v3/open/review/uplimit/hot` | 涨停梯队和热门题材 |
| `/open/sentiment/hot/today` | 当日情绪 |
| `/open/sentiment/hot/day` | 情绪历史 |
| `/open/sentiment/media/ths2/top` | 人气榜 |
| `/v3/market/trade/days` | 交易日历备用 |

不建议第一阶段依赖登录后的页面和 VIP 数据。登录接口可以保留，但不能作为核心链路。

### 4.4 采集任务设计

按时间分三类任务。

#### 盘前任务

时间：08:30 左右。

采集内容：

| 数据 | 来源 |
| --- | --- |
| 昨日复盘结论 | 本地数据库 |
| 隔夜新闻 | 财联社、东方财富、央视等 |
| 公告 | AkShare/东方财富 |
| 美股映射 | AkShare/腾讯等 |
| 今日交易日判断 | 扶摇交易日历 |

产出：

```text
盘前指引
今日观察题材
今日风险点
观察池
```

#### 盘中任务

时间：交易时段，每 30 到 60 秒轻量刷新，重点数据每 3 到 5 分钟聚合。

采集内容：

| 数据 | 来源 |
| --- | --- |
| 实时行情 | 扶摇 |
| 指数快照 | 扶摇或 AkShare |
| 涨跌家数 | 扶摇全市场快照聚合 |
| 观察池快照 | 扶摇 |
| 异动 | AkShare 或本地快照差分 |

产出：

```text
盘中首页
实时观察池
异动时间线
涨幅梯队
```

#### 盘后任务

时间：15:30 到 17:30。

采集内容：

| 数据 | 来源 |
| --- | --- |
| 涨停原因 | Quantzz |
| 涨停梯队 | Quantzz |
| 热门题材 | Quantzz |
| 情绪热度 | Quantzz |
| 跌停、炸板 | AkShare |
| 龙虎榜 | AkShare |
| 日 K | 扶摇 |
| 财务/成分股 | 扶摇，低频任务 |

产出：

```text
复盘首页
涨停复盘
情绪周期
赚钱效应
题材轮动
自动复盘报告
```

### 4.5 数据标准化

必须统一这些字段：

| 标准字段 | 说明 |
| --- | --- |
| `trade_date` | 交易日，格式 `YYYY-MM-DD` |
| `thscode` | 标准代码，例如 `600519.SH` |
| `ticker` | 纯代码，例如 `600519` |
| `stock_name` | 中文名 |
| `source` | 数据来源 |
| `source_updated_at` | 外部数据更新时间 |
| `fetched_at` | 本地抓取时间 |
| `raw_payload` | 原始返回，方便追查 |

统一代码后，前端和业务层尽量都用 `thscode` 做主键。

### 4.6 数据源健康检查

每次采集都记录：

| 字段 | 说明 |
| --- | --- |
| `source` | 数据源，例如 `fuyao`、`quantzz`、`akshare` |
| `endpoint` | 接口或任务名 |
| `started_at` | 开始时间 |
| `finished_at` | 结束时间 |
| `status` | `success`、`failed`、`partial` |
| `rows_count` | 入库数量 |
| `error_message` | 错误信息 |
| `request_id` | 外部接口返回的请求 ID，如有 |

数据总览页面要能看见：

```text
今天哪些数据成功了
哪些失败了
失败后用了哪个备用源
当前页面展示的是今天数据还是旧数据
```

## 5. 开发阶段

### 阶段一：复盘产品成型

目标：先做成一个完整可用的盘后复盘产品。

内容：

1. 整理首页总览。
2. 加强涨停复盘页面。
3. 加强情绪周期页面。
4. 做题材轮动独立页。
5. 做数据源健康检查。

验收：

```text
每天收盘后可以自动更新
页面能回答今天强弱、主线、风险、明日观察
数据失败时能看出来
```

### 阶段二：接入扶摇，补实时能力

目标：把基础行情层补稳。

内容：

1. 新增 `FuyaoClient`。
2. 标准化股票代码。
3. 新增实时行情接口。

## 6. 风险和边界

### 6.1 不建议做的事

| 事项 | 原因 |
| --- | --- |
| 直接复制 Quantzz 页面样式 | 有版权和品牌风险，也不利于做自己的产品 |
| 依赖登录后的 VIP 页面 | 稳定性差，权限不清晰 |
| 所有数据一开始都全市场入库 | 成本高，容易拖慢开发 |
| 一开始就做 WebSocket | 先用轮询足够，后面有必要再加 |
| 一开始迁 PostgreSQL | 先把产品和数据逻辑跑通 |

### 6.2 需要注意的数据风险

| 风险 | 处理 |
| --- | --- |
| 外部接口字段变化 | 保留 `raw_payload`，客户端做字段兼容 |
| 接口限流 | 加缓存、降频、分批 |
| 盘中数据延迟 | 页面显示更新时间 |
| 数据源失败 | 主备切换，页面提示 |
| 股票代码混乱 | 尽快标准化到 `thscode` |

## 7. 最小可行版本

如果要最快做出一个“像 Quantzz 一样能用”的版本，建议只做这些：

1. 首页总览。
2. 涨停复盘。
3. 情绪周期。
4. 题材轮动。
5. 盘前指引。
6. 数据源健康。
7. 扶摇实时行情接口。

先不做：

```text
复杂策略模型
财务分析
WebSocket
用户系统
多账户
移动端
```
